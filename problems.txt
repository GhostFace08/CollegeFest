BACKEND API ROUTES
=====================================
AUTH:
  POST /api/auth/login          → returns { token, user: { _id, name, email, role, isApproved, committees[] } }
  POST /api/auth/register       → student registration (auto-approved)
  POST /api/auth/register-admin → admin request (isApproved: false)
  GET  /api/auth/me             → returns full user with populated committees

EVENTS:
  GET    /api/events                    → public, approved only
  GET    /api/events/admin/mine         → admin's own events (filtered by createdBy)
  GET    /api/events/:id
  POST   /api/events                    → admin creates (status: draft)
  PUT    /api/events/:id                → admin edits (resets to draft)
  DELETE /api/events/:id
  POST   /api/events/:id/submit         → submit draft for approval

SUPERADMIN EVENTS:
  GET    /api/superadmin/events/pending       → pending events only
  PATCH  /api/superadmin/events/:id/approve
  PATCH  /api/superadmin/events/:id/reject    → body: { note }
  PATCH  /api/superadmin/events/:id/unpublish

REGISTRATIONS:
  POST   /api/registrations               → body: { eventId } — student registers
  GET    /api/registrations/my            → student's own registrations (populated with event+student)
  DELETE /api/registrations/:id           → cancel
  POST   /api/registrations/verify        → QR verify
  GET    /api/registrations/event/:eventId → admin sees who registered

VOLUNTEERS:
  GET    /api/volunteers
  POST   /api/volunteers                  → body: { name, email, phone }
  POST   /api/volunteers/:id/assign       → body: { event: eventId, role }
  PATCH  /api/volunteers/:id/attendance   → body: { assignmentId, attended }
  DELETE /api/volunteers/:id

FINANCE/SPONSORS:
  GET    /api/finance/sponsors
  POST   /api/finance/sponsors
  PUT    /api/finance/sponsors/:id
  DELETE /api/finance/sponsors/:id
  GET    /api/finance/event/:eventId
  PUT    /api/finance/event/:eventId

LOGISTICS:
  GET    /api/logistics/:eventId
  PUT    /api/logistics/:eventId          → body: { venue, notes, checklist: [{item, done}] }
  NOTE: checklist items use "item" field (not "task") in backend

SUPERADMIN:
  GET    /api/superadmin/dashboard
  GET    /api/superadmin/committees
  POST   /api/superadmin/committees
  PUT    /api/superadmin/committees/:id
  DELETE /api/superadmin/committees/:id
  GET    /api/superadmin/admins
  GET    /api/superadmin/admins/pending
  PATCH  /api/superadmin/admins/:id/approve     → body: { committeeIds: [] }
  DELETE /api/superadmin/admins/:id/reject
  PATCH  /api/superadmin/admins/:id/committees  → body: { committeeIds: [] }

KEY IMPLEMENTATION DETAILS
=====================================
localStorage keys:
  'collegefest_token' → JWT token
  'collegefest_user'  → JSON stringified AuthUser object

AuthUser shape:
  { id, name, email, role, isApproved, committees: [{_id, name}], committee: string (first committee name) }

fest-context normalizers:
  normalizeEvent     → maps _id to id, populates committee name from committee object
  normalizeReg       → maps _id to id, extracts event._id and student._id
  normalizeVolunteer → maps _id to id, committee name from committee object
  normalizeSponsor   → maps actionChecklist to checklist array
  normalizeLogistics → maps checklist[].item to task field
  normalizeAdmin     → maps isApproved bool to status ('approved'/'pending')

=====================================
KNOWN BUGS (all fixed before dockerization)
=====================================

BUG 1 — Admin can see/edit events from ALL committees (not just their own)
  Root cause: admin.events.tsx filters events as:
    const myEvents = events.filter(e => e.committee === user?.committee);
  But /events/admin/mine already returns only their events.
  Fix: In admin.events.tsx change:
    const myEvents = events.filter(e => e.committee === user?.committee);
  To:
    const myEvents = events;

BUG 2 — Admin can't see logistics, volunteers, or sponsors
  Root cause: Pages filter by committee name from mock data or API calls fail silently.
  Fix: Add useEffect in admin.logistics.tsx, admin.volunteers.tsx, admin.finance.tsx:
    const { volunteers, refreshVolunteers } = useFest();
    useEffect(() => { if (volunteers.length === 0) refreshVolunteers(); }, []);

BUG 3 — New events not appearing on superadmin side / not saving to DB
  Root cause 1: createEvent sends committee as a name string but backend expects ObjectId.
  Root cause 2: After createEvent, superadmin refreshEvents is not re-fetching.
  Fix in createEvent payload:
    committee: user?.committees?.[0]?._id || e.committee  ← use _id not name
  Fix in admin.events.tsx handleSave:
    After createEvent(), call refreshEvents() to reload.

BUG 4 — Admin dashboard shows all zeros
  Root cause: Stats computed from empty events array before data loads.
  Fix: In admin.dashboard.tsx add:
    const { events, loading, refreshEvents } = useFest();
    useEffect(() => { refreshEvents(); }, []);

BUG 5 — Student registrations not showing (My Registrations tab empty)
  Root cause: Frontend filters registrations by studentId but user.id from
  localStorage may not match student._id from DB.
  Fix: Remove frontend filter — API already returns only that student's registrations.
  Change:
    registrations.filter(r => r.studentId === user?.id)
  To:
    registrations
  Also add useEffect in student.registrations.tsx:
    const { registrations, refreshRegistrations } = useFest();
    useEffect(() => { refreshRegistrations(); }, []);

CLOUD DEPLOYMENT — GCP COMPUTE ENGINE
=====================================
PLATFORM DECISION:
  Chosen: GCP Compute Engine (Ubuntu 22.04, e2-micro, free tier — $300 credit)
  Reason: Full SSH access, supports Docker + Jenkins + Ansible on same/sibling VM
  Alternatives considered: AWS EC2, Azure VM, DigitalOcean, Railway, Render

GCP VM SETUP STEPS:
  1. Create project at console.cloud.google.com
  2. Compute Engine → VM Instances → Create Instance
       Machine type : e2-micro (free tier)
       OS           : Ubuntu 22.04 LTS
       Firewall     : allow HTTP + HTTPS
  3. VPC Network → Firewall → Add rule:
       Name    : allow-collegefest
       Source  : 0.0.0.0/0
       Ports   : TCP 8080, 5000
  4. SSH into VM via GCP browser console
  5. Install Docker:
       sudo apt update
       sudo apt install -y docker.io docker-compose
       sudo usermod -aG docker $USER
  6. Clone repo and run:
       git clone https://github.com/your-repo/CollegeFest.git
       cd CollegeFest
       docker-compose up --build -d
  7. Access app at http://<GCP-EXTERNAL-IP>:8080

IMPORTANT — VITE API URL:
  Frontend API calls currently point to localhost:5000.
  On GCP these must point to the external IP or Docker service name.
  Set VITE_API_URL=http://<GCP-IP>:5000 as a build arg in the frontend Dockerfile.
  Optionally configure Nginx to proxy /api → http://backend:5000/api
  so both frontend and API are served from port 8080.

STEP 9 — DOCKER SETUP
=====================================
backend/Dockerfile:
  FROM node:18-alpine
  WORKDIR /app
  COPY package*.json ./
  RUN npm install
  COPY . .
  EXPOSE 5000
  CMD ["node", "src/app.js"]

frontend/Dockerfile (two-stage build):
  FROM node:18-alpine AS build
  WORKDIR /app
  COPY package*.json ./
  RUN npm install
  COPY . .
  RUN npm run build

  FROM nginx:alpine
  COPY --from=build /app/dist /usr/share/nginx/html
  EXPOSE 8080
  CMD ["nginx", "-g", "daemon off;"]

docker-compose.yml (project root):
  version: '3.8'
  services:
    frontend:
      build: ./frontend
      ports:
        - "8080:8080"

      build: ./backend
      ports:
        - "5000:5000"
